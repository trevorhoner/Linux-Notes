# Guidelines & Nomenclature

**Linux Notes** promotes creative expression and freedom of thought in teaching Linux.  
We encourage contributors to write instructionals in their own teaching and learning style — no rigid templates required.  

**The guiding philosophy is simple:**  
"Make the guide you wish you had when learning Linux."

That said, for maximum clarity and ease of use (especially since everything must fit on **one page**), we suggest following a loose, logical structure:

- Break the topic into major **step sections** named clearly from beginning to end.  
  Examples for partitioning a used hard drive:  
  - Backup  
  - Resize  
  - Make (partitions)  
  - Format (filesystems)  
  - Mount  
  - Common errors / Troubleshooting  

- Under each section heading, keep content concise:  
  - 3–4 explanatory sentences (describe what, why, and any key concepts).  
  - Followed by numbered sub-steps: e.g., **Backup.1**, **Backup.2**, **Backup.3**...  
    Use these for commands, actions, and inline explanations.  

This pattern helps readers scroll predictably and find what they need fast, while leaving you free to adapt:  
- Rename or reorder sections as the topic requires (e.g., "Preparation" → "Backup", or add "Security Notes").  
- Merge small sections if they fit better together.  
- Include good redundancy — repeat helpful ideas (commands, concepts) where they aid understanding in context.  
- Use Markdown freely: code blocks, bold/italics for emphasis, warnings/notes, tables for comparisons.

The goal is a self-contained, scrollable page that's helpful and personal — not cookie-cutter. If your style deviates but still teaches clearly within one page, that's welcome. Questions? Edit the discussion on this page or open an issue.

Happy documenting!
